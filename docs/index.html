GithubPages Umlaut test

Symptoms

Ä Ä; Ö Ö; Ü Ü;
ä Ä; ö Ö; ü Ü; ß ß;
Here & Auml; is followed by the iso-8859-1 Ä; and so on.
Every well behaving (Apache 2.4 without forcing encoding) web server delivers this page unspoiled but Github Pages won't.
With GitHub Pages all iso-8859-1 characters above 127 get displayed as question mark in black diamond (�).

Background

Our very first (quasi hello world) experience with GitHub Pages was disappointing: Almost all well tested pages transferred (pushed) there were miss-rendered by several browsers. This page was started as test page to explore the failure's reason.

In spite of
  <?xml version="1.0" encoding="iso-8859-1"?>
and
  <meta http-equiv="Content-Type" content="text/html; charset=iso-8859-1" />
as well as even (irrelevant for character set settings, of course)
  <meta http-equiv="content-language" content="de" />
all German Umlauts are spoiled.

See this very page on Apache 2.4:
  weinert-automation.de/infos/umlautGitHubPages.html
And see it on GitHub Pages:
  a-weinert.github.io/umlautGitHubPages.html

Diagnosis

The failure has two reasons:
  1. GitHub Pages says encoding UTF8 in its response header — at least in our case. Searches, so far, found no documentation therefore. And all introductory examples — three steps to your first web page on GitHub — don't give a warning.
  2. There seems to be a rule that the response header's encoding indication overrules all header and xml attribute setting in a web page. And all browsers seem to obey this rule.
    Considering a perfectly consistent static web page encoded in iso-8859-11 and saying so, this behaviour respectively preference rule seems a totally senseless recipe for false rendering.
Tip: No web server should use encoding in its response header when (as usual) unaware of the content.
