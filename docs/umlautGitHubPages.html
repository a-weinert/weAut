GithubPages Umlaut test

Symptoms

Ä Ä; Ö Ö; Ü Ü;
ä Ä; ö Ö; ü Ü; ß ß;
Here & Auml; is followed by the iso-8859-1 Ä; and so on.
Every well behaving web server (Apache 2.4 without forcing encoding) delivers this page unspoiled but Github Pages won't.
With GitHub Pages all iso-8859-1 characters above 127 get displayed as question mark in black diamond (�).

Background

Our very first (quasi hello world) experience using pages generated by our development / deployment environment at GitHub Pages was disappointing: Almost all well tested pages transferred (pushed) there were miss-rendered by several browsers. This page was started as minimal test page to explore the failure's reason.

In spite of
  <?xml version="1.0" encoding="iso-8859-1"?>
and
  <meta http-equiv="Content-Type" content="text/html; charset=iso-8859-1" />
as well as even (irrelevant for character set settings, of course)
  <meta http-equiv="content-language" content="de" />
all German Umlauts French accents etc. are spoiled.

See this very page on Apache 2.4:
  weinert-automation.de/infos/umlautGitHubPages.html
And see it on GitHub Pages:
  a-weinert.github.io/weAut/umlautGitHubPages.html

Diagnosis

The failure has two reasons:
  1. GitHub Pages says encoding UTF8 in its response header — at least in our case. Searches, so far, found no documentation therefore (except Jekyll naming UTF-8 as default). And all introductory examples — three steps to your first web page on GitHub — don't give a warning.
  2. There seems to be a rule that the response header's encoding indication overrules all header and xml attribute setting in a web page. And all browsers seem to obey this rule.
    Considering a perfectly consistent static web page encoded in iso-8859-11 and saying so, this behaviour respectively preference rule seems a totally senseless recipe for false rendering.
Tip: No web server should use encoding in its response header when (as usual) unaware of the content.

No remedies

Best would be all browsers giving the explicit page encoding preference over the server response. The latter should only the (server region) default, when the page encoding was forgotten. But that cannot be changed.

If working in Western Europe and doing almost everything in ISO-8859-1 or similar code pages you would like to avoid UTF8. Theoretically GitHub pages should obey
encoding: ISO-8859-1
in the _config.yml. But that doesn't yield the expected result. And, most probably Jekyll, Liquid and Co. would not work when turning off UTF-8 — and they are gorgeous.
Hence, when going GitHub Pages turn your development environment to UTF-8, even when living and working in France, Italy, Netherlands, Belgium ...
